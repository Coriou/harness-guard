**Consequence for Task 19:** if confirmed, source code is a legitimate,
execution-free, primary-evidence source per
`docs/maintenance/runbook.md` (retrieval URL, retrieval date, semantic hash,
pinned commit/release — the same rigor already required for docs pages).
Reading source to establish config key names, defaults, version-detection
signal, and packaging/install-channel facts does not require running the
binary at all, and is arguably *stronger* evidence than docs prose for those
specific claims (docs can lag or omit; the parser/config-loader code cannot).

**What source reading can establish (execution-free):**
- The exact config surface (`~/.grok/config.toml` keys, or wherever the
  loader reads from) and their defaults, directly from the config-parsing
  code.
- The version-detection signal (binary name, `--version` string format, or
  package metadata) directly from the CLI entrypoint / build metadata.
- The packaging and install-channel facts (npm package name, GitHub release
  artifact naming, etc.) directly from the packaging manifests / release
  workflow in the repo.

**What source reading likely cannot establish alone:** any claim about
*runtime upload/telemetry behavior* — what the binary actually transmits,
when, and to where. That's a claim about executed behavior, and reading the
code that constructs a network request is suggestive but not equivalent to
observing it fire (dead code, feature flags, build-time conditionals, and
server-side behavior are all invisible to static reading). The clean-room
lab run's core value — the mitmproxy/tcpdump capture with canary-token
search — is about that behavior claim specifically, not the config surface.

**The next session must make this split explicit and batch it as an owner
decision before starting Task 19's Step 1:** likely reduced scope is "read
the source for config-surface/defaults/detection/packaging facts (no lab run
needed for these), then decide with the owner whether any remaining
behavior claim in the rule set still requires the full lab run, or whether
source-level evidence plus the existing risk research is sufficient for
0.0.1." Do not assume either direction — present the split and let the
owner choose, since this changes both the RELEASE-GATING blocker's shape and
possibly the wording of
`docs/research/protocols/grok-build-cleanroom.md` itself (which may need a
new "source-reading" evidence path added alongside the lab-run path, not a

## 4. Next direction after 0.0.1 (owner request — needs brainstorm-pipeline design, NOT ad-hoc implementation)

The owner wants to explore **harness "trails" observability** after 0.0.1
ships: conversation-history stats, token usage, and generally the most
360-degree view of each harness's local footprint that Harness Guard's
execution-free, no-network posture still permits.

**The hard tension to resolve at design time:** `CONTEXT.md`'s product
safety invariants forbid reading prompt/session transcript *contents*. A
naive "trails" feature that reads conversation history to report stats
would violate that invariant if it parses message bodies at all, even just
to count them, unless the counting can be proven not to require
interpreting content (e.g., counting JSON array entries in a transcript file
without ever reading the `content` field's string value might be
defensible; reading token counts a harness itself already recorded as
metadata is safer still).

The likely resolution — to design properly, not assume — is one of:
- **Metadata-only stats**: file counts, sizes, modification dates, growth
  rate/retention posture, derived without ever parsing message content
  (e.g., `wc -l`-style structural counts, byte sizes, timestamps).
- **A new, explicit, separately-consented tier**: a scan mode a user must
  opt into by name (not the default `scan`) that is explicit about reading
  more than the current allowlisted config keys, with its own safety
  invariant carve-out documented and reviewed.

This needs the full `brainstorm-pipeline` treatment (context-digester →
brainstorm-worker → spec-to-plan), not ad-hoc implementation, specifically
*because* the invariant tension above is a genuine design decision, not an
implementation detail. It also needs **per-harness fresh evidence** of what
"trails" (history files, token-usage logs, session indexes) each of the
three harnesses actually leaves on disk before any design can be concrete —
that evidence-gathering is itself probably the first research fan-out step
of the pipeline.

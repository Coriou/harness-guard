# Competitive Landscape — 2026-07-14

**Status:** Research artifact (verified links, retrieved 2026-07-14). Not application ground truth; re-verify before citing in rules.
**Scope:** Direct/adjacent competitors, vendor built-ins, content competitors, gap and threat analysis for Harness Guard.

Bottom line up front: **the exact niche (local-only, read-only, source-cited, individual-dev privacy/config-posture auditor) is genuinely unoccupied, but it is a thin sliver inside a crowded, fast-moving "AI agent security scanning" category, and one close analog already exists.** Do not present this as a blue ocean.

## Vendor built-ins

- **Claude Code `/doctor`** — confirmed via docs/search: diagnoses installation type, connection status, sandbox availability, permission config, and OpenTelemetry exporter errors. It reports whether the sandbox is *actually* protecting you (a real overlap point) but does **not** produce a scored/cited privacy-retention-telemetry-training report — it's a "why isn't this working" tool, not a "what does this expose" tool. [code.claude.com/docs/en/monitoring-usage](https://code.claude.com/docs/en/monitoring-usage)
- **Codex CLI `codex doctor`** — support-ready diagnostics across runtime, auth, terminal, network, config, local state; redacts raw values in JSON output. Same category: environment/setup health, not a posture-with-citations audit. [gradually.ai Codex CLI changelog](https://www.gradually.ai/en/changelogs/codex-cli/)
- **GitHub Copilot cloud agent config audit API** (new, 2026-05-18) — lets orgs audit a *repository's* Copilot cloud agent MCP config, tool grants, Actions policy, firewall config via REST API. This is fleet/org-facing and cloud-agent-specific, not local individual CLI posture. Still worth flagging: GitHub is visibly moving toward "audit our own agent config" as a first-party enterprise feature, which is the direction that could eventually commoditize a slice of this. [github.blog changelog](https://github.blog/changelog/2026-05-18-audit-repository-copilot-cloud-agent-configuration-via-the-rest-api/)
- No vendor currently ships a cross-tool, cited, known/unknown-labeled local privacy posture report. That gap is real today but narrowing at the edges (sandbox status in `/doctor`, redaction in `codex doctor`, Cisco's Watchdog tamper detection).

## Content/guide competitors

- **Voibe AI Privacy Tracker** — [getvoibe.com/resources/ai-privacy-tracker](https://www.getvoibe.com/resources/ai-privacy-tracker/) — This is the strongest single threat in the "satisfies the need without a tool" category. Covers 34 tools including Claude Code, GitHub Copilot, Cline, Cursor, Tabnine; scores Training/Retention/On-device/Track-record (0–25 each, 0–100 composite); **extensively footnoted with primary sources** (vendor policies, court filings, regulator docs), explicit "we don't paraphrase what the policy probably says" methodology, reviewed monthly with a dated changelog. It does not check *your* local machine config — it's vendor-policy-level, not your-actual-settings-level — but for a user just trying to decide "is this tool safe," it may fully satisfy the need Harness Guard targets, especially for people who haven't yet installed anything.
- **Augment Code privacy comparison page** — [augmentcode.com/tools/privacy-comparison-of-cloud-ai-coding-assistants](https://www.augmentcode.com/tools/privacy-comparison-of-cloud-ai-coding-assistants) — Not a real threat: vendor-biased content marketing, minimal sourcing, ends in an Augment sales pitch. Low quality but shows the format ("privacy comparison of coding assistants") is already being used for lead gen — exactly the play recommended in the project's own decision-and-strategy.md for benjsmin.com.
- **Tom's Guide "I checked the privacy settings of every major AI chatbot"** and similar consumer press pieces — chatbot-focused (ChatGPT/Gemini/Copilot/Grok), not CLI/coding-tool-specific, policy-review methodology rather than config inspection. Low direct overlap but shows mainstream press appetite for this exact framing.
- Net: static/policy-level comparison content is well covered and actively maintained (Voibe in particular is a legitimate, well-sourced competitor to the *information* need); nobody but claudit-sec inspects *your actual local files*.

## Gap analysis — is the niche actually empty?

Not empty, but real and narrow. Three things are simultaneously true:

1. **The "MCP/agent security scanner" category is crowded and well-funded** (Snyk, Cisco, Armor1, Lakera→Check Point, Prompt Security→SentinelOne, plus dozens of smaller OSS scanners). If Harness Guard is perceived as "another agent security scanner," it loses immediately on resources and breadth.
2. **The specific sub-niche — local-only, read-only, zero execution, zero network at scan time, explicit unknown-state labeling, per-finding primary-source citation, covering privacy/retention/telemetry/training/sync (not just attack-surface/prompt-injection) — has exactly one close analog (claudit-sec), and it's Claude-only, doesn't cite sources per finding, doesn't do telemetry/retention/training, and appears to have gone quiet for ~2 months.** No tool currently does this across Claude Code + Codex CLI + Copilot CLI together with the evidence-schema rigor described in decision-and-strategy.md (evidence class, valid_from/valid_until, confidence, source URL per rule).
3. **The content layer (Voibe tracker) already answers "should I trust this tool" reasonably well for people who haven't installed anything yet.** Harness Guard's actual differentiated value is for people who *already have these tools installed* and want to know what their specific local config currently does — a "did I actually turn that off" check rather than a "which tool should I pick" comparison. That's a real, defensible wedge, but it's narrower than "AI privacy" broadly and the plan's own framing (decision-and-strategy.md) already gets this right.

Verdict: proceed, but the credible claim is "the only local, execution-free, per-finding-cited config auditor spanning these three harnesses' privacy/retention/telemetry surface" — not "the only AI agent security tool." The plan document already avoids the second framing; the research confirms that's the correct restraint.

## Threat assessment — top 3 things most likely to eat this project's lunch

1. **A well-resourced adjacent player adds a "privacy mode" to an existing scanner** (likelihood: high). Snyk Agent Scan or Cisco's MCP Scanner are both under very active development (commits *today*, 2026-07-13) and both already do local multi-harness discovery. Adding telemetry/retention/training-toggle checks to an existing 2,778-star, daily-shipping tool is a much smaller lift for them than building distribution from zero is for Harness Guard. This is the single biggest risk — feature creep from an existing scanner, not a new entrant.
2. **claudit-sec (or a fork/successor) gets resourced and generalizes.** It already has the identical safety invariants (read-only, no network, Apache-2.0, redaction) and 290 stars with zero marketing. If HarmonicSecurity (or a fork) adds Codex CLI/Copilot CLI support, per-finding source citations, and telemetry/retention/training checks, it becomes functionally identical to Harness Guard's spec, with a head start. Currently stalled (~2 months no push) — a resourcing risk for *them*, but also means the window is open, not closed.
3. **The content layer wins the top-of-funnel and Harness Guard never gets discovered.** Voibe's tracker (and the inevitable "best AI privacy practices 2026" SEO content) already answers the question at the moment most users are searching for it — before they'd think to install a CLI tool. If Harness Guard doesn't own equivalent SEO/content real estate (which decision-and-strategy.md's benjsmin.com plan does address), it risks being technically superior but undiscovered next to content that "good enough"-solves the same anxiety.

Lower-likelihood but worth flagging: GitHub's move toward first-party Copilot agent-config audit APIs (2026-05-18) signals vendors are starting to build their own posture-audit surfaces for enterprise — a long-run commoditization risk for the fleet/team tier specifically, less so for the individual-dev CLI tier this project targets first.

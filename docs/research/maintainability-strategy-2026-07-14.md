# Maintainability Strategy Research — 2026-07-14

## Transferable maintenance patterns (per studied project, what applies)

**endoflife.date** — One Markdown-with-YAML-frontmatter file per product (`products/<name>.md`), validated live via a published `product-schema.json` (yaml-language-server + `bin/lint-product.sh`). The load-bearing mechanism: a companion `release-data` repo (git submodule) watched by Dependabot, which opens a PR "at most once a day" when upstream releases change, auto-updating version/date facts via per-product `auto:` config (npm registry, Docker Hub, git tags, etc.). Critically, **the bot never sets the EOL/judgment field** — only humans do that; automation surfaces candidates, never publishes verdicts. Also publishes a `/recommendations` page telling vendors how to publish EOL data durably (stable URL, explicit dates) — Harness Guard could publish an analogous "what we need from your docs" page aimed at Anthropic/OpenAI/GitHub.

**MDN browser-compat-data** — Schema structurally forces citation: `spec_url` is *mandatory* whenever a feature is marked `standard_track: true`. Two-tier governance (15 "Peers" self-merge data PRs after 1 peer/owner approval; 6 "Owners" gate schema/infra) scales review without bottlenecking one person. Freshness isn't manual doc-reading — `mdn-bcd-collector` auto-generates browser feature-detection tests, runs them via Selenium/BrowserStack on every new browser release since 2020, and diffs results against tracked data; Open Web Docs (funded nonprofit) reports this automates **~90%** of feature-addition/removal detection, leaving humans to interpret, not notice.

**caniuse** — One JSON file per feature (`features-json/`); compiled `data.json` is a generated artifact contributors are told never to hand-edit. New features land with `shown:false` and are flipped on only after maintainer validation — a clean "accepted but not yet trusted" staging pattern. Notably, caniuse itself leans on MDN BCD's automation ("data from BCD... updated weekly") rather than re-deriving support facts independently — evidence that even a long-running solo-adjacent project outsources fact-detection where it can. No public interview from Alexis Deveria on maintenance burden was found; funding is via Patreon.

**ToS;DR** — "Point" data model: atomic claims (`good/neutral/bad/blocker`, 0–100 score, plain-English tldr, `case` grouping for reusable identical judgments, `irrelevant`/`disputed` booleans + required `reason`) decoupled from the service they attach to — directly reusable for modeling individual Harness Guard config-key claims. Points are submitted by annotating a literal quoted span of the source document (via a Hypothesis-fork tool), not asserted in the abstract. No automated document-change-detection was found in ToS;DR itself — staleness handling is purely social/curator-driven. (A third-party project, TOS Tracker, does automated diffing of 45k+ legal docs on a rotating 2hr–multi-month schedule using ETag/conditional-request + content-diff — a usable independent proof-of-concept pattern.)

**privacyguides.org** — Recommendations structured as prose "Criteria" sections per category with explicit Minimum vs Best-Case bars (e.g., VPN's Security minimum requires a *published third-party audit* — citation-as-structural-gate, same idea as MDN's mandatory `spec_url`). Governance: any content change needs 2 trusted reviewers. No automated staleness tooling found — reactive/community-report-driven. Two real, citable incidents (Startpage delisted after adtech acquisition, Wire delisted after ownership change to a US holding co) are direct precedent for "vendor changed out from under a settled claim," which is exactly the Grok-mitigation-keys failure mode Harness Guard already hit.

**OSSF Scorecard** — Checks decomposed into small, independently-versioned "probes" with documented stability tiers — lets the project evolve individual heuristics without a full-check rewrite, good pattern for a solo reviewer's cognitive load. The real staleness tripwire isn't a bespoke detector — it's **per-check golden-fixture e2e test repos** that fail when the platform's actual behavior shifts underneath a check. Weekly self-scan cron explicitly drops expensive checks to control cost — direct precedent for trimming scope under a free-tier budget.

**Trivy / Grype vulnerability DB pipelines** — Both separate "fetch raw source" (stored in git, for diffability/audit trail) from "build queryable artifact" (published only after a smoke-test gate passes) into two distinct pipeline stages. Trivy's `trivy-db` cron runs every 6 hours and publishes to 3 registries (Docker Hub/GHCR/ECR Public) via ORAS as an OCI artifact; Grype's daily pipeline runs a smoke-test validation gate and refuses to publish a bad build. Both run far more compute than a solo dev has — the transferable lesson is the *shape* (raw-fetch → validate → publish, never publish unvalidated), scaled down to weekly/monthly and to git/GHCR instead of dedicated CDN infra.

**nvm / asdf** — Cheapest possible pattern: no polling infrastructure at all, just an on-demand fetch of a vendor-published machine-readable manifest (`nodejs.org/dist/index.tab`) triggered by user invocation. asdf plugins subcontract version-detection to a shared upstream `node-build` definitions repo rather than each reimplementing it — suggests a shared community "vendor config-key manifest" could eventually be worth seeding, though none of the 3 tier-1 harnesses currently publish anything like `index.tab`.

**Contribution-economics analogs (ToS;DR, Exodus/ETIP, Gitleaks, Semgrep, EasyList, Wappalyzer)** — see Contribution model section below.

## Recommended freshness pipeline (concrete: CI jobs, cadence, cost)

Free-tier feasible — **GitHub Actions minutes are unlimited on public repos** (no 2,000 min/month cap that applies to private repos), so a weekly/monthly job against ~50 URLs and 3 npm packages costs literally nothing against quota. Two real constraints to design around: (1) minimum sane cron interval is far looser than needed here — weekly/monthly is fine, don't go tighter; (2) **scheduled workflows auto-disable after 60 days with zero repo commits/PRs/issues** and need manual/API re-enabling — a real risk for a low-traffic solo project, needs either routine commit cadence or a documented manual-reenable step in the maintainer runbook.

Recommended shape, modeled directly on the endoflife.date / Trivy-Grype / OSSF Scorecard patterns above (raw-fetch stage separate from human-judgment stage, automation never publishes a verdict):

1. **Release-watch job** (see next section) — daily or weekly cron polling npm registry `dist-tags` for the 3 tracked packages, diffing against a git-committed last-seen JSON. On diff: open a triage issue (title includes tool + old/new version), link the changelog/release-notes diff. Cost: ~seconds of CI time, 3 HTTP GETs.
2. **Doc-drift job** — weekly cron over a maintained list of ~30–50 cited vendor doc URLs (the exact URLs Harness Guard's rules cite as sources). Use `lychee` first-pass for dead-link detection (proven GH Actions recipe: cron trigger → lychee-action → auto-open issue on failure via `peter-evans/create-issue-from-file`), plus a second content-hash pass modeled on `urlwatch`'s filter-before-hash approach (strip to semantic text before hashing, to avoid false positives from timestamps/nav/ads — every tool surveyed converges on this). On detected drift: fire a Wayback Machine "Save Page Now" (SPN2) API request to create a fresh archive.org snapshot, and link both old and new snapshots in the triage issue as the evidence anchor for re-review (directly modeled on the Perma.cc/InternetArchiveBot "detect → auto-archive → human resolves" pattern from legal-citation tooling, the closest real-world analog found for "cited source changed → flag the claim").
3. **Fixture/golden-test gate** — extend Harness Guard's existing fixture-test suite (already planned in the implementation plan) to serve double duty as the actual staleness tripwire, per OSSF Scorecard's pattern — a rule that silently stops matching real-world config shape is a stronger signal than a doc-hash diff for some claims.
4. **Human triage, never auto-publish** — all three jobs above only ever open an issue/PR with a diff; no automation is permitted to change a rule's verdict/status. This mirrors every project studied that has real automation (endoflife.date, MDN BCD) — the line between "surface a change candidate" and "certify a fact" is never crossed by a bot in any of them.

**Known pitfalls to design around up front** (from the doc-drift research): vendor docs are increasingly JS-rendered (Next.js/Docusaurus-style, which Anthropic/OpenAI/GitHub docs plausibly use) — a raw curl+hash may miss real changes or false-positive on hydration/build-ID churn; a Playwright fallback is heavier (CI minutes, bot-detection/WAF-blocking risk) and should be reserved for the handful of pages that actually need it, not the default. CDN edge-caching can also cause a CI runner to see a different response than a human browsing — not solvable, just worth expecting occasional false negatives/staggered detection.

## Release-watch design for the 3 tier-1 harnesses (concrete feeds/endpoints)

All three verified live via direct curl on 2026-07-14. Cheapest **uniform** trigger across all three: a single daily/weekly cron hitting the npm registry's abbreviated endpoint (header `Accept: application/vnd.npm.install-v1+json`, cheaper payload than the full doc) for all three packages, diffing the relevant `dist-tags` value against a committed last-seen file. GitHub `releases.atom` feeds are a good secondary signal for reading actual diff content once a bump is detected.

| Tool | npm package | Watch this dist-tag | GitHub releases.atom | Changelog file |
|---|---|---|---|---|
| Claude Code | `@anthropic-ai/claude-code` | `stable` (2.1.202) — not `latest`/`next`, which move ~daily and are noisy | `github.com/anthropics/claude-code/releases.atom` — live, tags match npm 1:1 | Real content: `CHANGELOG.md`, plain `## X.Y.Z` headers, diffable |
| Codex CLI | `@openai/codex` | `latest` only (0.144.4) — package has ~16 dist-tags total (per-platform alpha/beta/native channels), 3,329 total versions; everything else is noise | `github.com/openai/codex/releases.atom` — live but mixes alpha prereleases; filter out `-alpha.` tags if consuming directly | Stub — points to GitHub Releases page instead; use release body, not CHANGELOG.md |
| Copilot CLI | `@github/copilot` | `latest` (1.0.70), not `prerelease` (1.0.71-0) | `github.com/github/copilot-cli/releases.atom` — live | Real, large `changelog.md` (~4,900 lines), version-headed, diffable. Note: `github/copilot-cli` repo is a distribution/issue wrapper only (install.sh + docs) — the CLI itself is closed-source |

Registry endpoint pattern for all three: `https://registry.npmjs.org/{package}` (full) or same URL with the abbreviated `Accept` header (cheap). No auth required for any of the three.

## Rule versioning & auto-degradation recommendation

- **Encode tested version ranges explicitly per rule** (min/max verified version), in the spirit of MDN BCD's `version_added`/`version_removed` and Terraform's pessimistic `~>` operator (confirmed-safe-up-to-just-below-next-major). Where a rule is confirmed true from some version onward but the exact origin is uncertain, MDN's `≤`-prefix convention (`"≤79"` = "confirmed by this version, possibly earlier") is a clean, already-battle-tested way to encode that imprecision honestly rather than picking a fake-precise number.
- **Absence, not a sentinel, means unknown.** MDN's current schema deliberately dropped `true`/`null` as valid `version_added` values in favor of either a real version string or explicit `false` — where no data exists, the convention is to omit the entry rather than encode an ambiguous in-band "unknown." This matches Harness Guard's own stated `unknown` status but argues for implementing it as *absence of a matching rule-version-range entry*, not a magic value threaded through evaluation logic.
- **Explicitly choose and document which direction "unknown" degrades to, and test it.** This is the single most important, concretely evidenced finding from this stream: a real production bug (Vercel Turbopack / `vercel/next.js#92091`) occurred because a Rust `browserslist-rs` consumer let "unknown version" silently resolve to "support everything" (all-`None` interpreted as no constraints), which enabled compat transforms that then crashed. The fix explicitly flipped what "unknown" means and made it a named, tested flag rather than an implicit fallthrough. **CONTEXT.md already commits Harness Guard to reporting `unknown` rather than guessing — this research is a direct warning that the *implementation* of that principle needs an explicit, fixture-tested default (almost certainly "most conservative / flag for user attention," never "assume safe" or "assume risky"), because an untested implicit default can silently invert the intended safety property.**
- **Mandatory source citation as a schema constraint, not an editorial nicety** — mirror MDN's "spec_url required when standard_track:true" and Privacy Guides' "third-party audit required to hit the Security minimum bar." Harness Guard's rule schema should make `source_url` + `retrieved_date` structurally required whenever a rule's status is anything other than `unknown`, so the schema itself, not just review discipline, prevents another Grok-mitigation-keys-style stale/unsupported claim from entering the ruleset.
- **Use fixture/golden tests as the staleness tripwire** (OSSF Scorecard pattern) in addition to doc-hash monitoring — a rule silently failing to match real config shape is stronger evidence of drift than a hash change on a doc page that may not have changed the actual behavior.

## Contribution model recommendation (what to structure as data PRs)

**Structure as small, single-file, schema-validated data files — one rule/config-key claim per file** (endoflife.date's one-file-per-product, caniuse's one-file-per-feature, both explicitly forbidding hand-edits to any compiled/aggregate file). Concrete template fields, synthesized from the strongest analogs:
- Required, minimal: tool, config key/path, claimed behavior, `source_url`, `retrieved_date`, tested version (mirrors Exodus/ETIP's "only 2 required fields, everything else deferred" — their explicit lesson was that a low mandatory-field bar is what makes a template usable by non-experts).
- Optional/deferred: platform variations, plan/auth-context notes, related CVE/issue links.
- A `case`/grouping concept (from ToS;DR) if the same vendor behavior recurs across multiple claims, to avoid re-litigating an identical judgment N times.

**Two-tier trust model**: anyone can submit (issue or data-file PR), a second trusted reviewer approves — matches Privacy Guides' 2-reviewer gate and MDN's peer-approval model, and is cheap to bootstrap (needs exactly one trusted second person, not a team).

**Important caveat from real evidence, not assumption**: Exodus/ETIP's own maintainer commentary is explicit that even a near-zero-friction submission form did not remove the real cost — independent double-checking by at least two people, described as "tedious" and "almost invisible from the outside," is the actual bottleneck, not the submission format. **Budget reviewer time, not just submission-template design, as the real contribution cost.** EasyList's contrasting practice (they explicitly prefer issues over PRs for filter-list changes, because "the team will need to look through the sites anyway, and pull requests can fall victim to rebase errors") suggests Harness Guard may get more real signal, for less maintainer overhead, from a **structured issue template** (URL + before/after claim text) during the current data-curation-heavy phase, reserving the stricter schema-validated PR template for once the rule schema and fixture-test harness are mature enough that contributors can self-serve a passing PR.

**Data/core separation is a forkability insurance policy, evidenced concretely**: when Wappalyzer went closed-source/paywalled in 2023, its pre-paywall MIT-licensed JSON fingerprint files (already structurally separate from the product code) let community forks keep the OSS ruleset alive and updated. Contrast: CRXcavator (Cisco/Duo) was killed with zero warning and never open-sourced, so nothing could be forked — "abandoned but forkable" vs. "just gone" is entirely determined by whether data/core separation and permissive licensing exist *before* the crisis, not after. Harness Guard's rules should live in a clearly separate, permissively-licensed directory/package from day one (the implementation plan's `rules/` top-level directory already does this — worth treating as a hard invariant, not just a folder convention).

## Sustainability risks & mitigations (honest)

**Is monthly verification of 3 tools realistic solo? Conditionally yes — but only if the freshness pipeline above is actually built, not just aspirational. As a manual-only commitment, no.**

The evidence bounding this: Tidelift's maintainer survey found 58% of maintainers have quit (22%) or seriously considered quitting (36%); top reasons were competing life/work priorities (54%), loss of interest (51%), burnout (44%), insufficient compensation (38%, rising). Secondary (lower-confidence, not independently re-verified) figures from the same survey family suggest unpaid hobbyist maintainers on *popular* projects average roughly 9 hours/week, versus 20–30 hrs/week for paid professional maintainers. Harness Guard's maintainer is unpaid, solo, and has separate consulting work — realistically at or below the ~9hr/week hobbyist ceiling, likely less in any given month with active client work. **Manually re-deriving every claim for 3 vendors from scratch every month is not sustainable at that budget and will silently rot exactly as CONTEXT.md fears.**

What makes it realistic instead: shifting the maintainer's actual monthly task from "notice what changed" (expensive, unbounded) to "review a small number of automation-flagged diffs" (cheap, bounded) — the same shift Open Web Docs made for MDN BCD, where `mdn-bcd-collector` automates ~90% of feature-change detection and humans only interpret. If the release-watch and doc-drift jobs above are running, most weeks nothing fires; the maintainer's real cost becomes reviewing the handful of triage issues that do fire, plausibly on the order of low single-digit hours/month in steady state, spiking around major tool version releases. **This makes the freshness pipeline a precondition for the monthly-verification claim being honest, not an optional nice-to-have** — shipping the "verified monthly" SLA before the pipeline exists would be the kind of overclaim the verification audit already flagged once.

Concrete failure modes to explicitly design against, each with a real precedent found in this research:
- **Unbounded review backlog from accepted contributions.** Steve Francia (Cobra/Afero maintainer) reports a real security vulnerability sat unreviewed for months in Afero purely from backlog volume — not malice, just accumulated PR/issue volume outpacing review capacity. Mitigation: the strict, narrow contribution template above exists specifically to keep this bounded; be willing to close out-of-scope PRs fast (see the FastMCP maintainer's "Guide to Saying No" — keep a written scope statement specifically so declining is easy and non-negotiable, and explicitly name a niche-one-user-ask category as reject-by-default).
- **Burnout-driven maintainer handoff as an active security risk, not just a stall.** XZ Utils' Lasse Collin was verifiably overwhelmed and under a multi-year social-engineering campaign pressuring a co-maintainer handoff before the eventual backdoor. If/when Harness Guard ever needs a second maintainer, the handoff itself needs deliberate vetting — this is a security-relevant decision, not just a scheduling one, especially for a tool whose whole premise is trustworthiness.
- **Single point of failure in infrastructure, not just code.** privacytools.io's founder going unresponsive while solely controlling the domain forced the entire project to abandon its domain and rebrand (became Privacy Guides, 2021) rather than risk losing it. Mitigation: domain/npm-publish-rights/GitHub-org ownership should have a documented second trusted holder or succession plan from early on, independent of code contribution questions.
- **Abrupt, unforkable death is worse than honest retirement.** CRXcavator (Cisco) was killed overnight, never open-sourced, no migration path — total loss. Ingress NGINX's Nov 2025–Mar 2026 formal, publicly-announced retirement (after years of one/two volunteers sustaining ~50% of cloud-native ingress) is a legitimate, honorable outcome for a project that outgrew its maintainer capacity — and is explicitly compatible with CONTEXT.md's own phase-gated design ("if those conditions fail, publish the research/checklist and stop expanding"). Being genuinely OSS + local-only + permissively licensed from day one (already the plan) is the single biggest insurance policy here, per the Wappalyzer/CRXcavator contrast above.
- **A genuine differentiation opportunity, not a pattern to copy**: none of the projects studied (including the two closest in subject matter, ToS;DR and privacyguides.org) publicly display a "verified as of DATE" / "unverified past N days" staleness badge to end users — this looks like unclaimed territory rather than an established practice, and would directly operationalize CONTEXT.md's "explicit about limitations" invariant in a way users can see at a glance, at low implementation cost once the freshness pipeline's last-checked timestamps already exist as data.

**Bottom line**: solo + AI-leverage + monthly SLA on 3 tools is a reasonable target only if roughly the first 4 items under "Recommended freshness pipeline" ship before or alongside the first public release claim of "verified monthly" — treat the automation as scope-blocking for that specific claim, not a later enhancement.
